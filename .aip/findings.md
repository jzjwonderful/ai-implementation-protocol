# 侧发现台账（triage 收件箱）· append-only

> **用途**：开发任务 A 时撞见的、**与 A 无关的其他问题**（别的 bug、隐患、坏味道）的原始收件箱。**捕获，但别追**（capture, don't chase）——不让侧发现把当前任务带偏，也不让它静默流失。
> **它不是 STATUS**：本表是原始收件箱（证据齐就登记）；STATUS 已知缺口 / decisions.md / 新 feature 是**分类后的出口**。

## 发现即 3 秒分流（撞到规格外的东西就触发）
| 判定 | 出路 |
|---|---|
| **阻塞**当前任务正确性/验收 | 不是"侧"发现：并入本任务或叫停；实质改范围 → 停下来问用户 |
| 非阻塞 + **trivial 且同爆炸半径**（正在改的文件） | 顺手修，写进 commit message，**不扩规格、不进本表** |
| 非阻塞 + **非trivial / 跨域** | **登记本表一条，继续干当前任务** |

## 生命周期（收件箱 → 出口）
登记 → 分类去向之一：① 立项为新 feature；② 记 `decisions.md`（若是决策）；③ 进 `STATUS.md` 已知缺口（接受为债）；④ 关闭（won't-fix / 已被他改修掉）。
> review/check 闸门：本轮侧发现有无被静默丢弃？本表新增项是否都标了去向？有"待分类"残留 = 不放行。

## 格式
```
## F-N：<一句话症状>
- 发现 / 状态：YYYY-MM-DD（发现于<哪个任务>）/ 待分类 | 已立项 | 接受为债(STATUS #x) | 已修 | won't-fix
- 影响域：<按项目填，如 前端 | 服务端 | 客户端 | 契约 | 跨层>
- 初判严重度：高 | 中 | 低
- 症状：现象是什么
- 证据：repro / 日志路径·时间戳 / 端点 等（发现当时就抓，事后会蒸发）
- 去向：分类结论 + 链接（STATUS #x / ADR-N / 报告 / issue）
```
