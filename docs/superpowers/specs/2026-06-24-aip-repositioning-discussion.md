## 1. 对现状的诊断

AIP 当前把功能重心压在**「用会拦截的机器闸门防止 AI 交付中的漂移/遗漏/跑偏」**。
方向对，但有两处失衡：

1. **最硬的护城河（闸门）一半停在纸面**：整套「确定性闸门」的强度押在一个**尚未安装**的 pre-commit hook 上（STATUS 已知缺口第 1 条）。ADR-1 本身就是证据——主打防漂移的工具，自己漂出了协议。
2. **重「流程留痕」轻「判断质量」**：协议自己承认，方法质量（头脑风暴深不深、TDD 真不真）机器管不了，闸门只查「残留物」。后果是一份浅薄的工作也能跑出全绿的 `aip check`。闸门挡得住「静默遗漏」，挡不住「判断错误」。
3. 另有：双副本同步（顶层→plugins/）靠人工纪律，无机器闸门兜底（缺口第 2 条）；分发管线在抢设计注意力。

## 2. 用户的真实诉求（重新定位的种子）

- superpowers 是**任务级**的：对「把当前这件事做完」负责，做得好；但**不对项目负责**——纪律、风控、知识、复用这些跨任务、跨会话的东西，任务级 agent 天然不管也管不了。
- 由此 AI 会：重复造轮子 / 重复踩坑 / 跨会话有用信息断层 / 干 A 时发现 B 容易跑偏（finding 机制就是为此设计）。
- 用户想在 superpowers 之上补一个**项目级的「大脑」**：任务开始时从它读入（别人踩的坑、能复用的轮子、项目纪律），结束时把耐用信号写回。最近几个提交是这个目的，但**「用起来还没那么顺手」**。
- 更高目标：做一个**能自我迭代、自己成长**的系统。

## 3. 核心判断（第一性原理）

### 3.1 问题的根
AI 写代码有个绕不开的结构性事实：**上下文有界，工程无界**。每个会话只看得到工程的一个切片，工程却跨会话存在。工程越乱，痕迹越烂，AI 越要靠**猜**补空白——猜，就是重复造轮子/重复踩坑/跑偏的总根源。
真正稀缺的资源只有一个：**能跨会话存活、且加载成本低的可靠工程信息**。AIP 做的所有事，本质都是制造并保住这个。现在不顺手，是制造方式选错了。

### 3.2 判断一：少写，多派生
上下文进系统两条路：**人写**（质量高、有摩擦，乱工程里没人有纪律去写——连作者都漂了）vs **机器派生**（保真度低点、永远在线、零摩擦）。
老 AIP 几乎全押「人写」，这是它重、累、跑不起来的根因。**正确方向反过来：能机器派生的全派生，只让人写机器派生不出来的那一小撮。**
机器派生不出来的只有三样：
1. **为什么这么定**（decisions）
2. **验证过的真因/坑**（knowledge——机器没调过这个 bug，不知道根因）
3. **现在在做哪件事 / 卡在哪 / 下一步**（当前指针 + blocker）

### 3.3 判断二：自我成长 = 一个简单循环
不是玄学。就是：**每干完一个任务，把这次学到的教训自动记下来；下次碰到相关任务，自动提醒出来。**
现在的毛病：**记了（knowledge.md），但下次没在对的时候自动冒出来**，于是笔记躺成死文档。
再加一条：**同一个坑反复踩，就别只留提醒，加一条自动检查/拦一下**；从没用上的提醒就删掉。这样工程在它真栽过跟头的地方一点点变结实。
最难一关：**召回精度**——提醒太多变噪音（AI 干脆不看），太少坑照踩。「不顺手」的真名字就是这个。

### 3.4 统一视角（关键）
**「任务接管」和「知识库」是同一个问题**：
- 接管 = 这个**任务**的跨会话状态；
- 知识库 = 这个**项目**的跨任务状态。
两者都是「跨边界的状态怎么留」，答案一样：**只写不可派生的那一小撮 + 其余机器派生 + 在对的时刻召回。** 不该有两套机制、两种纪律。

## 4. 从 tech-ai-kits（用户另一套更成熟的系统）学到的

> 路径：`D:\my\my\tech-ai-kits-master`。它已经把上面的难点解掉一大半。

### 4.1 它有两套知识系统，分工不同
- **`.knowledge/`**（踩坑/评审规则）：markdown + **每次全读对账** + pre-commit hook + 跟 commit 绑死。**开发期用、零服务。** 条目格式固定：`现象/原因/规则/来源`（来源含 MR/issue + 日期 + 作者，全必填）。条目对应稳定 `rule_id`（snake_case），被 reviewer prompt 和 self-review 清单复用。
- **Kite KB**（packages/kite-kb + SQLite + FTS5）：运行期、要真搜（BM25 + kind 权重 + 时间衰减 + 相关度阈值）。AI 只能写 `draft`，用户在 Web 审核才变 `active + verified_by_user=1`、搜索才可见。

**关键洞察**：**召回策略要看库的大小**。库小 → 全读对账（不用搜索）；库大 → 真检索。AIP 现在的 `knowledge_index.md`（手工索引）卡在中间，所以最难受。

### 4.2 它解掉的三个难点（直接对应 §3.3）
1. **召回准**：`apply_scope` 字段（每条写清「什么场景用/不用」），命中后不在范围内当未命中 → 摁住「噪音」。
2. **杂物抽屉**：draft→人工审核才可见 → AI 灌不进垃圾。
3. **捕获时机**：做成工作流里的固定步骤（bug-analyzer Phase 7、提 MR 前 self-review 对账），不是额外作业；且**强制每次读全部 + 显式对账哪几条命中本次改动**。

### 4.3 除知识库外，值得搬进 AIP 的「纯方法、零依赖」机制
（来自 `skills/bug-analyzer/SKILL.md`，按价值排序）

1. **假设分级 + 强制门**：动手前列假设，每条标 **A 自己能查证 / B 只能问人 / C 背景项**。A 类必须自己先查不准问人；B 类没确认前不准写进主结论。→ 把「不许猜」变成可执行动作。**最该搬。**
2. **证据标签 + 整链置信度 = 最弱一环**：事实标 `[日志]/[代码]/[配置]/[已验证]` vs `[推断]/[假设]`；一条因果链有一环是推断，整个结论就不准当「高置信度」。→「内部自洽但前提全错的长篇推导」的解药。
3. **先横扫不漏方向，再深挖**：深挖前按固定维度清单扫一遍，每维标「相关/不相关/待查」，禁止此刻下结论。→ 防「一头扎进最先想到的那条链」（5why 最常翻车点）。
4. **力度分档**：lite/deepthink/auto + **solo（有人盯）/team（无人值守）**；无人值守时更保守（宁可「请补充信息」也不硬给结论、不擅自写入）。→ **直接治「小题大做/浪费时间」**，比 AIP 现在「一套重 feature 流程 + 一条 bug 轻轨」灵活。
5. **输出精简契约**：结论不是工作笔记、省过程细节、代码引用只留 1–3 处、无损精简到「最短能让人理解并行动」。→ 可直接抄成 AIP 写 handoff/知识条目的输出约束。
6. **（轻量）rule_id 串起 知识→自检→评审**：不用上机器 lint，让坑变成评审清单的一项就够 → 「软知识升级」的轻量版。

### 4.4 明确跳过的
带搜索的 Kite KB 服务端（要跑 FastAPI+SQLite，破坏 AIP「零依赖、像 .git 一样塞进任何仓库」的命）。TAPD/GitLab/CT 等都是 Kite 的业务，与 AIP 无关。

## 5. 已达成的决定

### 5.1 知识库走 A 方案（已定）
AIP 只抄 `.knowledge/` 那一套**轻量做法**：markdown + 每次全读对账 + 跟 commit 绑死 + pre-commit hook 兜底，再补上 `apply_scope` 字段和「draft 草稿→人工确认」。**不上服务端、不要搜索引擎。**
代价：知识条数涨到撑不住（参考 tech-ai-kits 的 `review-pitfalls.md` 已 596 行）时，「全读对账」会扛不住——那是「毕业到 Kite 那一档」的临界点，不该让 AIP 为此变重。
附带要求：**AI 总结必须简短扼要，不许废话连篇**（固定小模板 + 字数上限来强制）。

### 5.2 AIP 与 Kite 的关系（已认可）
- **AIP = 最低配**：任何仓库、零安装、只有文件和 git。最需要纪律的工程（乱、没基建、别人的 repo）恰恰起不了 Kite —— 这就是 AIP 的地盘。
- **Kite = 最高配**：有服务、能搜、Web 审核、跨项目共享。
- 有一条干净的升级线：先用 AIP 的「全读对账」，涨到撑不住再「毕业」到 Kite。
- **AIP 抽的是 Kite「开发期那一半」的原理，不是代码**；Kite 是领域专用，AIP 提供**通用空骨架**（由 `config.yaml` 的 lenses/真理源填充）。
- 风险红线：**别把 AIP 做成装在文件里的残废 Kite**。它的价值是「零依赖、随处可用」，凡是破坏这条的（如为搜索加后端）一律不要。

### 5.3 「无损接管」的结论（已认可）
不是「继续做厚」，而是**反过来做薄**：
- 大上下文（百万 token）已让**会话内**几乎不丢，handoff 只在**真正冷启动换会话**的边界才有用，而这没那么频繁。
- 接管要传的大部分能从 git diff 派生；推不出来的还是 §3.2 那三样。
- **「无损」是错的靶子**——接手的 AI 不需要完整回放，只要够用到「不重复干活、不重复踩坑」。核心机制应改述为「让所有 AI **以最小代价**接管」。
- 仍值得加强的一处：**跨工具接管（Claude↔Codex）**——大上下文解决不了，是 AIP「any AI」卖点的真正落点；但要传的也是最小内核，不是全套流水。

### 5.4 本次重构范围：砍 / 留 / 加（用户已定方向）
**砍**（per-feature 接管仪式整套去掉）：`features/<id>/` 下的 `handoff.md`、`task_board.yaml`、`session_log.md`、`verification.md`，及围着它们转的 `aip start/resume/done` 生命周期闸门。
**留**（活文档，AIP 真资产）：`STATUS.md`、`decisions.md`、`canonical-assets.md`、`findings.md`、`knowledge.md`。
**加**（Kite 开发期那一半 + 方法骨架）：
- `knowledge.md` 套 Kite 纪律：`apply_scope` 字段、固定条目格式、每次全读对账、AI 写草稿→人确认；
- root-cause skill 加方法骨架：假设分级、证据标签、横扫、输出精简、力度分档；
- `rule_id` 串起 知识→自检；pre-commit hook 当捕获触发点。

**Refinement（重要）**：「移除任务接管」≠ 移到零。唯一推不出来又不属于活文档的——「现在卡在哪 / 下一步 / blocker」——**并进 `STATUS.md`**（现有「当前焦点/在建」「已知缺口」两节已在装），删掉 per-feature 文件，不再单独成一套接管子系统。

**主动接受的代价**：删 per-feature `task_board` 就放弃了对「超大特性、跨很多次冷会话」的细粒度进度跟踪。判断值得（与「别按最坏情况设计」一致），但是主动取舍，不是白捡。

**连带后果**：`aip check` 换职责——从「接管完整性闸门」变成「活文档 + 知识纪律的卫生闸门」（知识索引一致、findings 已分类、知识条目有 apply_scope、STATUS 更新、无协议外并行产物）。它不消失，是换了份工作。
