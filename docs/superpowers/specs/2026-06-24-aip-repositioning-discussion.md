# AIP 重新定位设计 · 最终版

> **一句话定位**：AIP 是一个「管项目级状态和纪律」的轻量机制——六类活文档 + 自我成长引擎，零依赖、随处可用，命令由 AI 按时机自主触发，人日常只敲 `$aip init`。
> **语言铁律**：本文及一切由 AIP 产出的内容，一律大白话、结论先行，**禁止说黑话**（晦涩比喻、生造术语），公认技术名词除外。
> 配套实施计划：`docs/superpowers/plans/2026-06-25-aip-rebuild-core.md`。

---

## 1. 定位与要解决的问题

AI 写代码有个绕不开的事实：**每个会话只看得到工程的一个切片，工程却跨会话存在**。工程越乱、痕迹越烂，AI 越要靠猜补空白——猜，就是重复造轮子、重复踩坑、跑偏的总根源。

真正稀缺的资源只有一个：**能跨会话存活、且加载成本低的可靠工程信息**。AIP 做的所有事，本质都是制造并保住这个。

AIP 和 superpowers 这类框架互补：后者对「把当前这件事做完」负责，做得好，但不管跨任务、跨会话的纪律/知识/复用。AIP 补的就是这个项目级的空缺。

## 2. 设计原则

1. **少写多派生**：能从 git/代码免费派生的（改了哪些文件、过程流水账、谁改的）一律不手写；只手写机器派生不出来的那一小撮（为什么这么定、验证过的坑、现在卡在哪、领域概念）。
2. **自我成长 = 自动捕获 + 自动召回 + 周期纠偏**：干完一件事把耐用信号记下来，下次相关时自动冒出来，并在固定边界纠偏。最难的是召回精度。
3. **零依赖红线**：只靠文件和 git，像 `.git` 一样塞进任何仓库。凡破坏这条的（为搜索加后端等）一律不要。最需要纪律的工程（乱、没基建、别人的 repo）恰恰起不了重型工具，这就是 AIP 的地盘。
4. **命令 AI 自主**：人日常只敲 `$aip init`；其余 AIP 行为由 AI 按上下文自主触发，每次都吭一声让人能观测。
5. **禁止说黑话**：见上。

## 3. 文档模型（六类活文档）

### 3.1 总表

| 文档 | 装什么 | 管哪一面 | 时间朝向 |
|---|---|---|---|
| `OVERVIEW.md` | **手写多线看板**（顶部）+ **自动派生摘要**（下面） | 现状/接管入口，读它就够 | 往前看（看板）/ 派生 |
| `decisions.md` | 架构/方向级决策（非任务级需求） | 为什么这么定 | 回头看 |
| `knowledge.md`(+`_index`) | 验证过的技术坑/根因 | 踩过的坑 | 回头看 |
| `reference.md` | ①领域概念/术语口径 ②核心实体与关键铁律 ③可复用实现（裁决） | 这工程是什么 + 该复用什么 | 回头看 |
| `inbox.md` | 旁路问题收件箱 | 撞见、待分类的问题 | 待分类 |
| `conventions.md` | 代码风格/注释/设计风格/构建·调试·验收固定流程 | 怎么干 | 稳定规约 |
+ `config.yaml`（工程对接信息，零配置起步、靠捕获长大）、`protocols/`（协议本体）。

### 3.2 OVERVIEW（合并总览，取代独立 STATUS）

- 一页文档，分两段：**顶部「在建（多线看板）」是手写源**（AI 收尾时改），含 `▶[active]` 标记指明当前指哪条线；**下面「自动摘要」由 `aip overview` 从 decisions/knowledge/reference 派生**。
- 用标记块隔开手写区和自动区，自动重生绝不覆盖手写看板（`aip_discovery` 的托管块能力）。
- 冷启动/接手只读这一页：现在干啥 + 相关决策/坑/概念一眼全有。
- 多线看板：一条功能/bug 一块（卡哪/下一步/blocker/must_read）；大线外挂 `tracks/<id>.md`，看板块退化成一行指针。每块封顶几行；线 done 立即移出看板。
- 为什么不能删它：它装的「往前看 + 易腐」状态（卡在哪/下一步）是会话结束就永久蒸发、无法从 git 重建的，正是第 1 节那个稀缺资源本身。

### 3.3 各文档判据

- **decisions**：判据是高度，不是「不可逆」。只记架构/方向级（选型、模块边界、依赖取舍、协议规则）；单个任务的需求讨论/调整不进，做完即弃。
- **knowledge**：技术性为主——版本兼容、平台/工具链怪癖、API 实际行为、构建/环境坑，以及修 bug 踩的坑（根因+规则）。判据：下次/下个人会重复踩、且重新发现很贵。**只收已验证的**，猜测不进。条目格式：`现象/根因/证据/适用范围/状态/最后复核/关联`，索引含「适用范围」列以便按场景召回。
- **reference**：杂乱工程最缺、AI 最易瞎猜的就是领域模型（这系统在干嘛、核心概念是什么）。比「别重复造工具函数」重要得多。可复用实现那节只记裁决（钦定用哪个+边界），枚举类交给外部工具按需查。
- **inbox**：干 A 时撞见的、与 A 无关的问题。投递走第 4 节的检索去重，不无脑 append。
- **conventions**：项目「怎么干」的常驻规则；机器能强制的（linter/formatter/CI）指向它，文档只记需人/AI 判断的。

## 4. 自我成长引擎（三个跨文档机制）

- **M1 写入前检索去重**（inbox/knowledge/decisions）：投递前读全表、AI 判断有无类似 → 合并/加关联 or 新增；不无脑 append。旁路问题完整流程：撞到 → 先在 knowledge+inbox 搜（有无解决方案/旧讨论）→ 无则确认是新问题、整理后投 inbox。decisions 投递时顺带做「与历史决策冲突/重复」检测。
- **M2 完成时纠偏，不做周期性**：绑在「一条线 done」这个自然边界强制一次。**两档防臃肿**：①每次必做（便宜近自动）= 机器卫生（check）+ 只整理本次碰过的条目及其直接关联；②条件触发才全本语义纠偏（新增条目超 N / 检测到矛盾 / 距上次全本纠偏超 M 次完成）。**铁律**：纠偏只产 draft/diff，**绝不静默改**；过时条目 supersede 不删；每次纠偏是可审 commit。
- **M3 关联 + 反向链接**：正向手写 `关联:`（ADR-N / K-NNN / inbox#N / conventions#x）；反向由脚本派生（谁引用了我），不手写。任何稳定文档互引互查。

**诚实边界**：闸门只保证「整理/捕获动作发生了、没留机器可查的脏」，**保证不了语义质量**——质量靠 AI 认真 + 人抽查。自动纠偏是双刃：会把对的悄悄合并/删错，所以铁律是只产 draft/diff、supersede 不删、可审 commit。

## 5. 自动捕获

- **两路**：①当下顺手捕获（查清根因/撞见旁路/被纠正风格的那一刻就记，证据新鲜、质量最高，但会漏）；②完成闸门强制回扫（一条线 done 时强制问：学到什么进 knowledge？旁路问题进 inbox？工程怎么构建测试进 config？新概念/规约进 reference/conventions？）——兜底捞回漏的。长会话会丢上下文，所以趁新鲜就写，闸门只兜底。
- **统一写入纪律**：拟草稿 → 读目标文档比对去重（M1）→ 写 `状态: draft` → 吭一声。draft 直到人确认才 active；从不擅自当权威、不静默改旧内容。
- **机器 vs AI**：机器逼时机（钩子/闸门）+ 校验形式（check 字段齐、索引同步）；AI 供内容（捕获什么、像不像、是不是真因）。这和 AIP 一贯分法一致：机器管得了动作发生没、有没有脏，管不了判断对不对。
- **防噪音**：trivial + 同文件 = 顺手修不登记；只收已验证的进 knowledge。
- **防走过场**：回扫要列「本次具体碰过/学到的候选」，不是答「无」就过；闸门便宜 + 吭声可见。

## 6. 命令与触发

- **命令塌缩为 AI 自主**：人日常只敲 `$aip init`（把 AIP 装进一个工程，是人的明确决定）。其余行为（开线/投 inbox/沉淀/check/完成闸门/读 OVERVIEW 续作）由 AI 按自然时机自主触发。保留手动入口当备用（可选）。
- **自动触发的可靠性来自三层，不是 AI 记性**：
  1. **钩子**（pre-commit → check，机器强制、跳不过、装一次永久生效）；
  2. **完成闸门**（done 时强制纠偏+捕获回扫，兜底）；
  3. **会话开场引导**（每会话加载，让 AI 知道有 AIP、去哪找、何时读）。
- **吭一声**：AI 每次用 AIP 动作都简短报给人（记了什么、参考了哪条 knowledge、跑了什么闸门）。既让人掌控，也是观测 AIP 是否有用的唯一窗口。

## 7. init 与零配置自长

- **目标**：让一个原本没有 AIP 的工程，从此具备「跨会话/跨人可靠接管 + 自我沉淀纠偏」，并把 AIP 和这个工程对接好。
- **做四件事**：建 `.aip/` 拷活文档骨架 / 建空 `config.yaml` / 装钩子 / 写引导块。
- **默认不问工程的事（零配置）**：杂乱工程的主人自己也不知道、问的还是听不懂的词。`config`/`reference`/`conventions` 从空白靠捕获长大——用到时先看仓库自动认（有 `package.json` 就试 `npm test`…），认不准用大白话问一次、然后记下，下次不再问。连工程对接信息本身都是 AIP 自己攒出来的，正好就是自我迭代。
- 代价（诚实）：早期它笨一点，会问「这项目怎么构建/测试」。自动认可能认错 → 认了要跟人确认，不瞎猜。

## 8. 钩子 / 开场引导 / 输出语言铁律

- **钩子项目级，非用户级**：脚本进 `.git/hooks/pre-commit`（每仓库一份，不污染其他仓库）跑 `aip check`，红了挡住提交。能被 `--no-verify` 绕过，是第一道闸门、不是唯一防线（完成闸门和人兜底）。
- **开场引导只指路、不喂状态**：告诉会话「这儿有 AIP、去 `.aip/` 找什么、开始/接手任务前才读 OVERVIEW」，不把当前状态塞进上下文。
- **禁止说黑话（通用铁律）**：写进协议正文 + aip 技能本体（随分发生效）+ init 引导块指向；同时已加进用户本机全局规则。

## 9. 外部工具退化链

- AIP 本体只声明**能力意图**（改前查引用、造前查同类），**不点名具体工具**，**不探查/记录本机装了什么**（平台每会话已给可用清单，用时现场挑；用法知识由各工具自己的 skill 承载）。
- 退化链（写进协议动作，不靠 AI 自觉）：
  - 改前查引用：有 LSP 用 `findReferences`/`incomingCalls`，没有用 grep；
  - 造前查同类：有 nexus-query/CodeGraph 这类 skill 就用，没有用 grep + 读代码 + 查 reference 裁决。
- 分档：地板档 = `LSP（机会主义）+ grep + AI 读 + 裁决记录`，零增强够用；增强档（大/乱工程）才接 CodeGraph（实时全量符号图，专吃「传递影响/谁调谁」，Macsa LaserPrint QT 项目实测 13967 节点、~1s 实时，碾压快照）+ nexus-mapper 语义层；**产物放 `.codegraph/`、`.nexus-map/`，与 `.aip/` 解耦**，不在场就退回地板档。
- 两个子问题要分清：①「改了炸谁」= 影响/调用图，LSP（浅层）/CodeGraph（传递）解；②「同义造轮子」= 按语义找相似，LSP/CodeGraph 都解不了，只能 AI 读候选 + 查 reference 裁决。

## 10. AIP 与 Kite 的关系

- **AIP = 地板档**：任何仓库、零安装、只有文件和 git。
- **Kite = 加强档**：有服务、能搜、Web 审核、跨项目共享。
- AIP 抽的是 Kite「开发期那一半」的原理（`.knowledge` 式的全读对账纪律），不是代码、不是带搜索的服务端。
- 升级线：AIP 的知识库涨到「全读对账」扛不住时，再上 Kite 那一档；AIP 不为此变重。
- 红线：别把 AIP 做成装在文件里的残废 Kite——它的价值是零依赖、随处可用。

## 11. 落地节奏与里程碑

- **一期**（`docs/superpowers/plans/2026-06-25-aip-rebuild-core.md`）：新文档集 + OVERVIEW 派生 + check 新闸门 + knowledge schema + 零配置 init + 钩子 + aip 技能（命令自主 + 捕获纪律 + 退化链 + 禁止黑话）+ 协议/引导 + sync。里程碑：**v1 装回本仓库自托管**（第一次真正的自我迭代）。
- **重构方式**：完全重构、不背历史包袱（git 仍可回溯）。teardown 已在分支 `chore/aip-teardown` 推平旧 per-feature 引擎。

## 12. 二期与未决

- **二期**（建议用装好的 v1 自己来管）：M2 全本语义纠偏、M3 反链派生、自动捕获回扫的完整编排、使用统计（每条 knowledge 记命中/复用次数 → 清没用的）、归档（失效/老条目挪走、活文档保持小）、召回质量打磨。
- **未决**（待拍板）：方法骨架（假设分级 / 证据标签 / 鱼骨横扫）做成每次强制还是仅指导；力度分档（lite/deepthink、solo/team）怎么映射到 AIP 并与 superpowers 的 subagent 对接。
