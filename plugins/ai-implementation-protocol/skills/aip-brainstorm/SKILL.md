# 多 AI 文档讨论（brainstorm）

多个终端各开一个 AI，围绕同一个**议题文档**接力讨论。AI 之间不直接通信，一切观点、用户插话、结论都只通过文档传递。文档落 `.aip/brainstorm/<议题>.md`。

**分工**：AI 负责读文档、想观点、写文本；**文档的一切改动只调脚本** `python <engine>/scripts/aip_brainstorm.py`（`<engine>` 含义见 aip 技能），脚本确定性维护：谁能发言（`当前轮到`）、轮次计数、收敛判定、状态检查。**禁止**用编辑工具直接改议题文档。

## 你的角色（看用户怎么叫你）

- **发起者**：用户说"发起议题 X" → 走下面 `start`。
- **讨论者**：用户说"去监控/参与议题文档" → 走 `status` + `say`。
- 不确定就先跑 `status` 看文档状态，它会告诉你当前轮到谁。

## 子命令怎么用

**1. 发起议题（发起者，只做一次）**

```
python <engine>/scripts/aip_brainstorm.py start \
  --topic "议题标题" --as 甲 --participants 甲,乙 \
  --opening "你的开场立场全文" --stance 继续
```

- `--participants` 逗号分隔，**第一个是发起者**；代号甲乙丙或用户指定的名字。
- `--max-rounds N` 可选，默认 5（收敛保险丝，见下）。
- 开场立场算发起者的轮次 1 发言，发完自动轮到下一位。
- 建完告诉用户：文档路径、参与者代号、"现在去另一个终端让乙参与"。

**2. 参与讨论（讨论者，每次被叫来都走这两步）**

```
python <engine>/scripts/aip_brainstorm.py status --doc <议题或路径> --as 乙
```

- 输出：状态 / 轮次 / 当前轮到谁（带 `--as` 会标明是否轮到你）/ 本轮各立场 / 用户插话数 / 结构检查。
- **没轮到你**：原样汇报状态，停下。**禁止**替别人发言、禁止空等轮询。
- **状态不是 discussing**（converged/need-user/aborted）：汇报结果，停下。
- **轮到你**：先用 Read 通读文档全文（所有发言 + 用户插话），想好观点再发言：

```
python <engine>/scripts/aip_brainstorm.py say --doc <议题> --as 乙 \
  --text "你的观点全文" --stance 继续
```

**3. `--stance`（立场）三选一，每次发言必填，如实填写**

- `继续`：还有话说、还没共识。
- `同意收敛`：你认为当前讨论内容已足够形成结论。**全员**最新立场都是它 → 脚本自动把状态转 `converged`。
- `需要用户裁决`：出现 AI 们定不了的分歧 → 脚本**立即**把状态转 `need-user`，不等本轮走完。

**4. 用户插话（任何 AI、任何时刻，最高优先级）**

用户在某个 AI 会话里说了与议题相关的话（尤其是直接输入的观点/决定），听到的那个 AI **必须当场**记进文档，这是用户输入同步给其他 AI 的唯一通道：

```
python <engine>/scripts/aip_brainstorm.py note --doc <议题> --text "用户原话或忠实转述"
```

- 不占轮次、不改 `当前轮到`。
- 状态是 need-user 时用户给了答案 → 加 `--resume`，记录的同时恢复讨论。

**5. 收尾**

- 状态转 `converged` 后，收尾的 AI 先读全文，再 `conclude --text "共识点 + 行动项"` 写入结论；随后按 AIP 捕获纪律把结论沉淀进 `decisions.md` / `knowledge.md`（够格才写）。
- 状态转 `need-user` 后，用 `escalate --text "待裁决问题清单（列清选项和分歧点）"`，然后明确告诉用户"议题需要你裁决，问题已写进文档"。
- 轮次超过上限仍未共识，脚本自动转 need-user（保险丝，防止无限讨论），同样走 escalate。
- 用户要终止 → `abort --reason "原因"`。

## "监控"是怎么回事（对用户要诚实）

AI 不是守护进程，不会自己盯着文件。两种驱动方式：

1. **默认**：用户在各终端之间传话——"轮到你了，去看议题文档"。零依赖，任何 CLI 都行。
2. 所在 CLI 支持定时唤醒时（如 kimi 的 cron），可以给自己定一个几分钟后重跑 `status` 的任务；醒来后没轮到就停，**不连续空转**。

## 纪律

- 每次被调用，先用一句话向用户汇报：状态 / 轮次 / 当前轮到谁 / 你刚做了什么。
- 发言要读完全文再写，复述别人的观点不算发言；引用用户插话要标明是用户说的。
- 脚本是读-改-写，没有文件锁：**同一时刻只让一个终端在写文档**（发言或记插话），另一个终端等它做完再动；用户传话本身就是这个串行节奏。
- `status` 报结构有问题时，把问题原样告诉用户，不擅自修文档。
- 大白话，禁止黑话；`--text` 里也是。
